## Abstract

This RFC defines a comprehensive documentation strategy for WorldForge,
covering auto-generated API references, Python SDK documentation, architecture
guides, provider integration guides, tutorials, example applications, and a
unified documentation website. The goal is to make WorldForge accessible to
three audiences: application developers using the API/SDK, provider authors
integrating new world models, and operators deploying and managing WorldForge.

---

## Motivation

WorldForge currently has no documentation beyond inline code comments and a
minimal README. This is a critical barrier to adoption:

1. **No API reference**: The REST API has 27+ endpoints with no documentation.
   Users must read the Rust source code to understand request/response formats.
   The server implementation spans ~9,588 lines in a single file, making this
   particularly painful.

2. **No SDK documentation**: The Python SDK (worldforge-py) has no usage guide,
   no API reference, and no examples. Users discover functionality through
   tab-completion and trial-and-error.

3. **No architecture guide**: The relationship between crates (worldforge-core,
   worldforge-server, worldforge-providers, worldforge-verify, worldforge-py)
   is unclear. New contributors cannot orient themselves in the codebase.

4. **No provider integration guide**: Third-party developers who want to add
   new world model providers have no guidance. The provider trait has implicit
   contracts and conventions that are only documented in existing implementations.

5. **No tutorials**: There are no step-by-step guides for common tasks. Users
   must assemble workflows from API calls with no guidance on best practices.

6. **No changelog**: Users have no way to track changes between versions.
   Breaking changes are discovered by accident.

7. **Poor README**: The top-level README is a placeholder with minimal
   information. It does not convey the project's purpose, capabilities, or
   how to get started.

Without documentation, WorldForge cannot grow beyond its core developers.
This RFC addresses every gap systematically.

### 2. API Reference (Auto-Generated)

#### 2.1 OpenAPI-Based Reference

After RFC-0010 (API Hardening) is implemented, the REST API will have an
OpenAPI 3.1 specification auto-generated by utoipa. The documentation site
will render this spec using one of:

- **Stoplight Elements**: Modern, interactive API documentation renderer
- **Redoc**: Clean, three-panel API documentation
- **Scalar**: Next-generation API reference with try-it-out functionality

The chosen renderer will be embedded in the documentation website as a
React/Web component.

#### 2.2 API Reference Content

For each endpoint, the reference will include:

- HTTP method and path
- Description and use cases
- Request parameters (path, query, header, body) with types and validation
- Request body schema with examples
- Response schemas for all status codes (200, 400, 404, 422, 500)
- Code examples in Python, Rust, curl, and JavaScript
- Rate limiting information
- Related endpoints

Example structure for a single endpoint:

```markdown
## Create Prediction

`POST /v1/predictions`

Submit a prediction request to a world model provider.

### Request Body

| Field       | Type   | Required | Description                          |
|-------------|--------|----------|--------------------------------------|

| model       | string | Yes      | Model identifier                     |
| input       | object | Yes      | Input data (provider-specific)       |
| guardrails  | object | No       | Guardrail configuration              |

#### 2.3 Rust Crate Documentation (cargo doc)

Each crate will have comprehensive rustdoc documentation:

```rust
//! # worldforge-core
//!
//! Core types and traits for the WorldForge world model framework.
//!
//! This crate defines the fundamental abstractions used across all WorldForge
//! components:
//!
//! - [`Provider`] — trait for world model provider implementations
//! - [`PredictionRequest`] / [`PredictionResponse`] — prediction I/O types
//! - [`ModelCapabilities`] — what a model can do (video, point cloud, etc.)
//! - [`GuardrailConfig`] — safety check configuration
//!
//! ## Example
//!
//! ```rust
//! use worldforge_core::{Provider, PredictionRequest};
//!
//! async fn make_prediction(provider: &dyn Provider, request: PredictionRequest) {
//!     let response = provider.predict(request).await.unwrap();
//!     println!("Generated {} frames", response.output.frames.len());
//! }
//! ```
```

Documentation standards for all public items:

- Every public type, trait, function, and module has a doc comment
- Every doc comment includes at least one code example
- Examples are tested via `cargo test --doc`
- Cross-references use `[`backtick links`]` to related items
- Error conditions are documented with `# Errors` sections
- Panics are documented with `# Panics` sections (and avoided where possible)
- Safety requirements for unsafe code documented with `# Safety` sections

### 4. Architecture Guide

#### 4.1 System Architecture Document

A comprehensive architecture document covering:

```markdown
# WorldForge Architecture

## Overview

WorldForge is a unified framework for interacting with world model providers.
It abstracts the differences between providers (NVIDIA Cosmos, Google Genie 2,
Wayve GAIA, etc.) behind a common interface, adds safety guardrails, and
provides cryptographic verification of predictions.

## Crate Structure

```
worldforge/
├── crates/
│   ├── worldforge-core/      # Core types, traits, and abstractions
│   ├── worldforge-providers/  # Provider implementations
│   │   ├── cosmos/            # NVIDIA Cosmos provider
│   │   ├── genie2/            # Google Genie 2 provider
│   │   ├── wayve/             # Wayve GAIA provider
│   │   └── ...                # Other providers
│   ├── worldforge-server/     # REST API server
│   ├── worldforge-verify/     # ZK verification
│   └── worldforge-py/         # Python SDK (PyO3/maturin)
```

## Data Flow

1. Client sends prediction request via REST API or Python SDK
2. Server validates request and applies guardrail configuration
3. Server dispatches request to appropriate provider
4. Provider returns prediction output
5. Guardrail runner evaluates safety checks
6. (Optional) ZK proofs generated for verification
7. Response returned to client

### Provider Trait
The central abstraction. Every world model provider implements this trait...

### Prediction Pipeline
The prediction pipeline orchestrates the flow from request to response...

### Guardrail Runner
Safety checks applied to every prediction...
```

### 5. Provider Integration Guide

#### 5.1 Target Audience

Third-party developers who want to add support for a new world model provider
to WorldForge. This guide assumes familiarity with Rust and the provider's API.

#### 5.2 Guide Structure

```markdown
# Provider Integration Guide

## 1. Provider Trait Overview

Every provider must implement the `Provider` trait:

```rust
#[async_trait]
pub trait Provider: Send + Sync {
    /// Unique identifier for this provider (e.g., "cosmos", "genie2")
    fn id(&self) -> &str;

    /// Human-readable name
    fn name(&self) -> &str;

    /// List available models
    async fn list_models(&self) -> Result<Vec<ModelInfo>>;

    /// Get capabilities for a specific model
    async fn capabilities(&self, model_id: &str) -> Result<ModelCapabilities>;

    /// Run a prediction
    async fn predict(&self, request: PredictionRequest) -> Result<PredictionResponse>;

    /// Check provider health/availability
    async fn health_check(&self) -> Result<ProviderHealth>;
}
```

### Step 1: Create the Provider Crate
### Step 2: Implement Authentication

### Step 5: Implement Output Normalization
### Step 6: Register the Provider
### Step 7: Write Tests

## 3. Best Practices

- Normalize all outputs to WorldForge's canonical format

- Return structured errors, not string messages
- Support cancellation via CancellationToken

In this tutorial, you'll install WorldForge, start the server, and generate

export COSMOS_API_KEY="your-key-here"

```python
import worldforge

client = worldforge.Client("http://localhost:8080")

# Upload a scene image
prediction = client.predict(
    provider="cosmos",
    model="cosmos-7b",
    input={
        "image": "path/to/scene.jpg",
        "action": "move_forward",
        "num_frames": 16,
    },
)

# View the predicted future frames
prediction.save_video("output.mp4")
print(f"Generated {len(prediction.frames)} frames in {prediction.elapsed_ms}ms")
```

## Step 5: Apply Safety Guardrails

```python
prediction = client.predict(
    provider="cosmos",
    model="cosmos-7b",
    input={"image": "scene.jpg", "action": "move_forward"},
    guardrails={
        "physical_plausibility": {"max_acceleration": 10.0},
        "temporal_consistency": {"threshold": 0.95},
    },
)

for check in prediction.guardrail_results:
    print(f"  {check.name}: {'PASSED' if check.passed else 'FAILED'}")
```

- Integrated with cargo-release for crate publishing
- Python SDK changelog generated separately for PyPI releases

### 11. Documentation CI/CD Integration

#### 11.1 Documentation Build Pipeline

```yaml
# .github/workflows/docs.yml
name: Documentation

on:
  push:
    branches: [main]
    paths:
      - 'docs/**'
      - 'crates/*/src/**'
      - 'openapi/**'
  pull_request:
    paths:
      - 'docs/**'

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Build Rust docs
      - name: Build cargo doc
        run: cargo doc --no-deps --all-features

      # Build Python SDK docs
      - name: Build mkdocs
        run: |
          pip install mkdocs-material mkdocstrings[python]
          cd docs/python-sdk && mkdocs build

      # Build Docusaurus site
      - name: Build Docusaurus
        run: |
          cd docs/website
          npm ci
          npm run build

      # Deploy to GitHub Pages
      - name: Deploy
        if: github.ref == 'refs/heads/main'
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./docs/website/build

  # Link checker
  link-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Check links
        uses: lycheeverse/lychee-action@v1
        with:
          args: --verbose --no-progress 'docs/**/*.md' 'README.md'
          fail: true
```

#### 11.2 Documentation Quality Checks

- **Link checker**: Verify all internal and external links on every PR
- **Spell checker**: cspell for technical documentation
- **Vale linter**: Enforce writing style (WorldForge style guide)
- **Code example testing**: All code blocks marked with language tags
  are extracted and tested
- **OpenAPI spec validation**: Validate that the spec is valid OpenAPI 3.1
- **Screenshot testing**: Visual regression tests for documentation pages

## Open Questions

1. **Docusaurus vs. mdBook**: mdBook is Rust-native and simpler, but lacks
   the plugin ecosystem and multi-language support of Docusaurus. Should we
   reconsider mdBook for simplicity?

2. **Documentation Versioning**: Should we version documentation per crate
   version or per "release train" (all crates versioned together)?

3. **Interactive Examples**: Should tutorials include interactive code
   playgrounds (e.g., embedded Jupyter notebooks or WASM-based Rust
   playground)? This adds complexity but improves learning experience.

4. **Video Tutorials**: Should we create video versions of the tutorials?
   Video content has higher engagement but is expensive to maintain as the
   API evolves.

5. **Localization**: Should we plan for documentation translation (e.g.,
   Chinese, Japanese, Korean)? Docusaurus supports i18n natively, but
   maintaining translations is significant ongoing work.

6. **API Reference Format**: Stoplight Elements, Redoc, or Scalar for
   rendering the OpenAPI spec? Each has different trade-offs in
   interactivity, customization, and bundle size.

7. **Blog**: Should the Docusaurus blog be used for release announcements,
   technical deep-dives, and community showcases? Or should we use a
   separate platform (e.g., dev.to, Medium)?

8. **Documentation Review Process**: Should all documentation changes
   require review from a technical writer, or can any contributor merge
   docs changes?
