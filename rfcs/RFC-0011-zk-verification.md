## Abstract

This RFC defines the integration of zero-knowledge proof systems into
WorldForge to provide cryptographic verification guarantees for two critical
properties: (1) that a world model provider actually produced a claimed
prediction output (inference proofs via EZKL), and (2) that all safety
guardrails passed for a given prediction (compliance proofs via Cairo/STARK).
The design covers proof generation, verification, on-chain anchoring, and
integration with the existing worldforge-verify crate.

---

## Motivation

WorldForge aggregates predictions from multiple world model providers. In
safety-critical applications (autonomous driving, robotics, industrial
control), consumers of these predictions need more than trust — they need
cryptographic proof that:

1. **Inference Integrity**: The prediction was actually generated by the claimed
   model with the claimed inputs. Without this, a malicious or compromised
   provider could substitute arbitrary outputs. In autonomous driving, a forged
   prediction could lead to catastrophic planning decisions.

2. **Guardrail Compliance**: All configured safety checks (physical plausibility,
   consistency, ethical constraints) were actually evaluated and passed. Without
   this, a system could claim "all checks passed" while silently skipping
   expensive safety validations.

3. **Audit Trail**: Regulatory frameworks (EU AI Act, NHTSA ADS guidelines)
   increasingly require demonstrable proof that AI systems followed prescribed
   safety protocols. ZK proofs provide a compact, independently verifiable
   audit artifact.

4. **Multi-Party Trust**: When WorldForge operates as a cloud service (RFC-0014),
   the service operator, the provider, and the consumer are separate trust
   domains. ZK proofs eliminate the need to trust any single party.

The current worldforge-verify crate (1,581 lines) defines proof types and
traits but contains only a mock verifier that accepts all proofs. This RFC
replaces the mock with production ZK proof systems.

### What ZK Proofs Do NOT Provide

It is equally important to state what is out of scope:

- **Model correctness**: A proof that a model produced an output says nothing
  about whether that output is accurate, useful, or safe. The model itself
  could be fundamentally flawed.
- **Input authenticity**: Proofs verify computation on given inputs but do not
  prove the inputs are authentic sensor data (that requires separate attestation).
- **Real-time safety**: Proof generation adds latency. ZK proofs are for
  post-hoc verification and audit, not real-time safety enforcement.
- **Privacy of model weights**: EZKL proofs reveal the model architecture
  (circuit structure). If model weights are proprietary, additional measures
  (e.g., trusted execution environments) are needed.

### 2. EZKL Integration for Inference Proofs

#### 2.1 What EZKL Provides

EZKL (https://ezkl.xyz) converts neural network models (ONNX format) into
arithmetic circuits that can generate zero-knowledge proofs of inference. Given:

- A model M (ONNX graph)
- An input X
- An output Y

EZKL generates a proof π such that any verifier can confirm that Y = M(X)
without re-executing the model. The proof is succinct (constant size regardless
of model complexity) and verification is fast (milliseconds).

#### 2.2 Circuit Compilation (Offline, One-Time)

Before proofs can be generated, each model must be compiled into a circuit.
This is a one-time, expensive operation:

```rust
pub struct EzklCircuitConfig {
    /// Path to the ONNX model file

    /// Quantization bits for fixed-point arithmetic (lower = faster, less accurate)

    /// Automatically calculated if None.
```

### 3. Cairo/STARK Integration for Guardrail Compliance Proofs

#### 3.1 Why Cairo/STARK for Guardrails

While EZKL is optimized for neural network arithmetic, guardrail checks are
general computations (range checks, consistency logic, rule evaluation). Cairo
is a general-purpose ZK language that compiles to STARK proofs, making it ideal
for proving arbitrary computation. STARKs also have the advantage of being
post-quantum secure and requiring no trusted setup.

#### 3.2 Guardrail Circuit Design

Each guardrail check will be implemented as a Cairo program:

```cairo
// physical_plausibility.cairo
// Proves that a prediction satisfies physical plausibility constraints

#[derive(Drop)]
struct PhysicalPlausibilityInput {
    // Predicted positions (quantized to fixed-point)
    positions: Array<(i64, i64, i64)>,
    // Predicted velocities
    velocities: Array<(i64, i64, i64)>,
    // Time step between frames (milliseconds)
    dt_ms: u64,
    // Maximum allowed acceleration (m/s^2, quantized)
    max_acceleration: u64,
    // Maximum allowed velocity (m/s, quantized)
    max_velocity: u64,
}

#[derive(Drop)]
struct PhysicalPlausibilityOutput {
    passed: bool,
    max_observed_acceleration: u64,
    max_observed_velocity: u64,
    num_frames_checked: u32,
}

fn check_physical_plausibility(input: PhysicalPlausibilityInput) -> PhysicalPlausibilityOutput {
    let mut max_accel: u64 = 0;
    let mut max_vel: u64 = 0;
    let mut passed = true;

    let mut i: u32 = 1;
    loop {
        if i >= input.positions.len() {
            break;
        }

        // Calculate velocity between frames
        let vel = compute_velocity(
            *input.positions[i - 1],
            *input.positions[i],
            input.dt_ms
        );

        let vel_magnitude = magnitude(vel);
        if vel_magnitude > max_vel {
            max_vel = vel_magnitude;
        }

        if vel_magnitude > input.max_velocity {
            passed = false;
        }

        // Calculate acceleration
        if i >= 2 {
            let prev_vel = compute_velocity(
                *input.positions[i - 2],
                *input.positions[i - 1],
                input.dt_ms
            );
            let accel = compute_acceleration(prev_vel, vel, input.dt_ms);
            let accel_magnitude = magnitude(accel);

            if accel_magnitude > max_accel {
                max_accel = accel_magnitude;
            }
            if accel_magnitude > input.max_acceleration {
                passed = false;
            }
        }

        i += 1;
    };

    PhysicalPlausibilityOutput {
        passed,
        max_observed_acceleration: max_accel,
        max_observed_velocity: max_vel,
        num_frames_checked: i,
    }
}
```

#### 3.4 Supported Guardrail Checks as Cairo Programs

| Check                      | Cairo Program                | Inputs                          | Outputs                    |
|----------------------------|-----------------------------|---------------------------------|---------------------------|
| Physical plausibility      | `physical_plausibility`     | positions, velocities, limits   | passed, max values        |
| Temporal consistency       | `temporal_consistency`      | frame pairs, similarity thresh  | passed, divergence score  |
| Object conservation        | `object_conservation`       | object counts per frame         | passed, violations count  |
| Boundary compliance        | `boundary_compliance`       | positions, allowed regions      | passed, out-of-bounds pts |
| Energy conservation        | `energy_conservation`       | kinetic/potential per frame     | passed, max delta         |
| Collision detection        | `collision_detection`       | bounding boxes per frame        | passed, collision count   |
| Output range validation    | `output_range`              | raw outputs, valid ranges       | passed, out-of-range vals |

### 4. Proof Generation Performance

#### 4.1 Performance Targets

| Proof Type                 | Target Gen Time | Target Proof Size | Verification Time |
|---------------------------|----------------|-------------------|-------------------|
| Inference (small model)   | < 30 seconds   | < 50 KB           | < 10 ms           |
| Inference (medium model)  | < 5 minutes    | < 100 KB          | < 10 ms           |
| Inference (large model)   | < 30 minutes   | < 200 KB          | < 10 ms           |
| Single guardrail check    | < 5 seconds    | < 20 KB           | < 5 ms            |
| Aggregated guardrails (7) | < 30 seconds   | < 50 KB           | < 10 ms           |

#### 4.2 Proof Size Estimates

EZKL proofs using KZG commitments:
- Proof: ~2-10 KB depending on circuit size
- Public inputs: variable, typically 1-5 KB
- Verification key: ~1-2 KB
- Total per-prediction: ~10-50 KB

STARK proofs (Cairo):
- Proof: ~50-200 KB (STARKs are larger than SNARKs)
- Public inputs: ~1-5 KB
- Verification: no trusted setup needed
- Total per-guardrail-suite: ~50-100 KB

#### 4.3 Optimization Strategies

1. **Proof Generation Parallelism**: Inference proofs and guardrail proofs are
   independent and can be generated in parallel on separate CPU cores.

2. **Circuit Caching**: Compiled circuits are cached on disk. Re-compilation
   only needed when the model changes.

3. **Batch Proving**: Multiple predictions for the same model can be batched
   into a single proof (amortizing the fixed cost).

4. **GPU Acceleration**: EZKL supports GPU-accelerated proof generation via
   CUDA. This reduces inference proof time by 5-10x.

5. **Recursive Proof Composition**: Multiple proofs can be recursively composed
   into a single proof, reducing on-chain verification cost.

#### 5.2 Starknet Verification

Cairo/STARK proofs verify natively on Starknet:

```rust
pub struct StarknetVerifier {
    /// Starknet RPC endpoint
    pub rpc_url: String,

    /// Verifier contract address
    pub contract_address: FieldElement,

    /// Account for submitting verification transactions
    pub account: StarknetAccount,
}

impl StarknetVerifier {
    pub async fn verify_guardrail_proof(
        &self,
        proof: &GuardrailComplianceProof,
    ) -> Result<VerificationResult> {
        // Submit proof to Starknet verifier contract
        let call = FunctionCall {
            contract_address: self.contract_address,
            entry_point_selector: selector!("verify_guardrail_compliance"),
            calldata: encode_proof_calldata(proof),
        };

        let tx_hash = self.account.execute(vec![call]).send().await?;

        Ok(VerificationResult {
            verified: true,
            chain: Chain::Starknet,
            tx_hash: tx_hash.to_string(),
            block_number: None, // Filled after confirmation
        })
    }
}
```

Starknet integration:
- STARK proofs verify natively (no pairing operations needed)
- Verification cost: ~50,000 gas (significantly cheaper than Ethereum)
- Native composability with other Cairo programs
- Recursive verification supported

An on-chain registry tracks all verified predictions:

```solidity
contract WorldForgeVerificationRegistry {
    struct VerificationRecord {
        bytes32 predictionId;
        bytes32 modelHash;
        bytes32 inputHash;
        bytes32 outputHash;
        bool inferenceVerified;
        bool guardrailsVerified;
        uint256 timestamp;
        address verifier;

    mapping(bytes32 => VerificationRecord) public records;

    event PredictionVerified(
        bytes32 indexed predictionId,
        bytes32 indexed modelHash,
        bool inferenceVerified,
        bool guardrailsVerified,
        uint256 timestamp
    );

    function recordVerification(
        bytes32 predictionId,
        bytes32 modelHash,
        bytes32 inputHash,
        bytes32 outputHash,
        bool inferenceVerified,
        bool guardrailsVerified
    ) external onlyAuthorizedVerifier {
        records[predictionId] = VerificationRecord({
            predictionId: predictionId,
            modelHash: modelHash,
            inputHash: inputHash,
            outputHash: outputHash,
            inferenceVerified: inferenceVerified,
            guardrailsVerified: guardrailsVerified,
            timestamp: block.timestamp,
            verifier: msg.sender
        });

        emit PredictionVerified(
            predictionId,
            modelHash,
            inferenceVerified,
            guardrailsVerified,
            block.timestamp
        );
    }
}
```

### 6. Security Properties Analysis

#### 6.1 Properties Guaranteed by ZK Proofs

| Property | Inference Proof (EZKL) | Guardrail Proof (STARK) |
|----------|----------------------|------------------------|
| Computational integrity | Yes — output is correct for given model + input | Yes — all checks were executed correctly |
| Soundness | Yes — false proofs computationally infeasible (2^-128) | Yes — false proofs computationally infeasible (2^-80) |
| Completeness | Yes — valid computations always produce valid proofs | Yes — valid computations always produce valid proofs |
| Zero-knowledge | Configurable — inputs can be private or public | Configurable — check details can be private |
| Succinctness | Yes — proof size constant regardless of model size | Quasi-linear — proof size O(log^2 n) |
| Non-interactivity | Yes — Fiat-Shamir transform | Yes — Fiat-Shamir transform |

#### 6.2 Properties NOT Guaranteed

| Non-Property | Explanation |
|-------------|-------------|
| Model correctness | Proving M(X) = Y says nothing about whether M is a good model |
| Input authenticity | Proofs don't verify that inputs came from real sensors |
| Data freshness | Proofs don't prevent replay of old valid predictions |
| Availability | Proof systems don't guarantee the prover is available |
| Model privacy | EZKL circuit structure reveals model architecture |
| Liveness | Cannot force a provider to generate proofs |

#### 6.3 Threat Model

The ZK verification system protects against:

1. **Malicious provider substituting outputs**: A provider cannot claim model M
   produced output Y if it actually produced Y'. The proof would not verify.

2. **Skipped guardrail checks**: A system cannot claim all guardrails passed
   without actually running them. The STARK proof encodes the computation.

3. **Tampered audit logs**: On-chain verification records are immutable.
   A compromised server cannot retroactively change verification results.

4. **Man-in-the-middle modification**: Proofs are bound to specific
   input/output pairs. Modifying the prediction in transit invalidates the proof.

The system does NOT protect against:

1. **Compromised model weights**: If a model is backdoored, it will produce
   backdoored outputs, and the proof will correctly verify those outputs.

2. **Collusion**: If the provider and the WorldForge operator collude, they
   can generate valid proofs for any computation they jointly control.

3. **Side-channel attacks**: Proof generation timing may leak information
   about private inputs.

### Property-Based Tests

- For any valid model + input + output: proof generation succeeds and
  verification succeeds (completeness)
- For any valid proof: modifying any byte of the proof causes verification
  to fail (soundness, probabilistic)
- For any valid proof: modifying any public input causes verification
  to fail (binding)

### Performance Tests

- Proof generation time vs. model size regression tests
- Memory usage during proof generation
- Concurrent proof generation throughput
- GPU vs. CPU proof generation comparison

### Security Tests

- Attempt to generate valid proof for incorrect output (must fail)
- Attempt to verify proof with wrong verification key (must fail)
- Attempt to replay old proof for new prediction (must fail)
- Fuzz proof generation with random/malformed inputs

## Open Questions

1. **EZKL Maturity**: EZKL is actively developed but not yet stable (pre-1.0).
   Should we pin to a specific commit, or accept the risk of breaking changes?
   Alternative: Implement our own Halo2 circuits for critical models.

2. **Proof Storage**: Where should generated proofs be stored? Options:
   - In the prediction response (increases response size by 10-200 KB)
   - In a separate proof store (requires additional infrastructure)
   - On IPFS/Arweave (decentralized, but adds latency)

3. **Cost Model for On-Chain Verification**: At $5-15 per Ethereum verification,
   on-chain verification is expensive. Should it be:
   - Per-prediction (expensive, full auditability)
   - Batched (aggregate proofs every N predictions)
   - On-demand (only when a dispute arises)
   - Starknet-only (50-100x cheaper)

4. **Model Privacy**: Some providers may not want to expose ONNX models for
   circuit compilation. How should we handle this? Options:
   - Provider runs their own EZKL prover
   - Trusted execution environment (TEE) for circuit compilation
   - Accept that some providers cannot support inference proofs

5. **Quantization Accuracy**: EZKL quantizes floating-point models to fixed-point
   arithmetic. The quantized model may produce slightly different outputs than
   the original. What tolerance is acceptable? How do we communicate this to
   users?

6. **Regulatory Acceptance**: Will regulatory bodies (NHTSA, EU AI Act enforcers)
   accept ZK proofs as sufficient evidence of compliance? This is uncharted
   territory — we should engage with regulators early.

7. **Proof Composition**: Should we support recursive proof composition
   (proving that a proof is valid)? This enables constant-size proofs for
   batches of predictions but adds implementation complexity.

8. **Cairo vs. Noir vs. Circom**: We chose Cairo for guardrail proofs. Should
   we also evaluate Noir (Aztec) and Circom for specific use cases? Cairo
   has the advantage of native Starknet deployment, but Noir has better
   Ethereum tooling.
